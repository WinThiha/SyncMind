## Context

Currently, the Docker containers run as the default user (usually `root` or `www-data` with UID 33). When these containers write to shared volumes (like `backend/storage/logs/`), the files are owned by those internal UIDs. Since the host developer usually has UID 1000, they lack the permissions to edit or delete these files from their host machine without using `sudo`.

## Goals / Non-Goals

**Goals:**
- Sync the UID/GID of the container processes with the host developer (UID 1000).
- Allow the developer to edit/delete any file generated by the application (logs, cache, generated code).
- Keep the configuration portable so it works for developers with different UIDs (via `.env`).

**Non-Goals:**
- Applying these changes to production environments.
- Completely rebuilding Docker images for non-root users (leveraging existing images).

## Decisions

- **Dynamic User ID Mapping**: Use `${PUID:-1000}:${PGID:-1000}` in the `user:` field of `docker-compose.yml` for both `backend` and `frontend` services.
- **Environment Configuration**: Add `PUID` and `PGID` to `backend/.env.example` and `frontend/.env.example`.
- **Permission Reconciliation**: Provide a manual one-time command for developers to reclaim existing files on the host that are currently owned by the container UIDs.
- **Entrypoint Compatibility**: Ensure `backend/docker-entrypoint.sh` doesn't fail when run as a non-root user (e.g., bypassing `chown` calls if permissions are already correct).

## Risks / Trade-offs

- **Risk**: Some containerized tools (like `composer`) expect the user to have a entry in `/etc/passwd` or a valid `$HOME`.
  - **Mitigation**: Set the `HOME` environment variable to `/tmp` or another writable directory if the mapped user is "unknown" to the container's internal OS.
- **Risk**: Running as a non-root user prevents the `docker-entrypoint.sh` from fixing permissions itself.
  - **Mitigation**: Rely on the developer to fix host-side permissions once, and thereafter the container (running as them) will create files they own.
